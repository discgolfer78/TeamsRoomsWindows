The mission & the team

In Part 1 we deployed an Android-based Microsoft Teams Rooms device. In Part 2 we do the same for a Windows-based MTR — and this time we lean on Windows Autopilot + Autologin so the device provisions itself, touchless, the moment it's plugged in onsite.

The destination is InfoComm 2026 — the AV/UC industry's flagship show. We're deploying two Windows-based MTR devices: the first is built on the Shure IntelliMix Foundation Compute (IMXCPU5) mini-PC, and the second is a Logitech Teams Rooms on Windows system. There's a nice bit of history here: Microsoft actually announced Autopilot and Autologin for Teams Rooms on Windows as generally available at InfoComm, so using it to deploy at InfoComm 2026 feels right at home.

Two people, one seamless deployment

The whole point of this architecture is that the heavy lifting happens in the cloud before the hardware ever reaches the show floor. That splits the work cleanly between two roles:

Cloud prerequisites (Mark's checklist)

Before any hardware ships, I confirm the tenant-side prerequisites for Windows Autopilot + Autologin. Most of these are configured once and reused for every future room.

  • Teams Rooms Pro licenses — one per device, assigned to each resource account. Critically, Teams Rooms Pro includes the Intune and Microsoft Entra ID P1 licenses needed for Autopilot. (Teams Rooms Basic will not work with Autopilot.)
  • Permissions — for Intune: Intune Administrator or Policy and Profile Manager; for the Teams Rooms Pro portal: Teams Rooms Pro Manager. Microsoft recommends least-privilege roles over Global Admin.
  • Automatic Intune (MDM) enrollment is set up, and devices Microsoft Entra join (self-deploying mode does not support Entra hybrid join).
  • Windows 11 on a certified MTR device with TPM 2.0 (required for self-deploying mode attestation).
  • Resource accounts created and licensed (we automate this in Section 11).
Teams Rooms Pro Intune Entra ID P1 Windows 11 TPM 2.0 Entra join

1 Register the device in Partner Center

Autopilot needs the device's hardware identity loaded into the tenant. As a CSP, I register the Shure and Logitech Windows based MTR devices through Microsoft Partner Center using the PKID from the box.

  • Register the device with PKID 555555555555 in Partner Center.
  • Set the Group Tag to MTR-Shure_Conference_Room_Infocomm2026. The MTR- prefix is mandatory — it's how the Teams Rooms Pro portal distinguishes Teams Rooms from every other Autopilot device, and it's what the dynamic group keys on.
⚠️
The MTR- Group Tag is not optional

When loading the hardware identity, Microsoft requires the GroupTag to be prefixed with MTR-. Get this wrong and the device won't be recognized by the Pro Management portal or picked up by the dynamic device group.

Docs: Partner registration · Autopilot registration for partners

2 Create an Entra ID dynamic device group

I create one dynamic device group — MTR Dynamic device group — and target all MTR policy at it: apps, the Autopilot profile, the ESP, and LAPS. When a device is registered in Partner Center, with the MTR- Group Tag, this dynamic membership rule automatically adds the MTR device to the group. Note: you may need to sync in Intune and it can take up to 15-20 mins for the deviec to show up in Intune.

Dynamic membership rule Entra ID
(device.devicePhysicalIds -any _ -startswith "[OrderID]:MTR-")
☁️
One group to rule them all

Targeting everything at a single dynamic device group keeps the deployment clean and repeatable. If you also manage non-MTR Windows devices in the same tenant, Microsoft recommends excluding this group from your other policies so the rooms only get the policies meant for them.

Docs: Create a group for the Teams Rooms devices

3 Deploy the MTRP Provisioning Tool

The Teams Rooms app update tool (MTRP Provisioning Tool) updates the Teams Rooms application to a version that supports Autopilot + Autologin. I package it as a Win32 app in Intune and assign it to the MTR Dynamic device group, and include it in the ESP so it runs during provisioning.

  • Created the app: MTRP Provisioning Tool (created June 8, 2026, right before the InfoComm event in Las Vegas).
  • App type: Windows app (Win32), assigned to MTR Dynamic device group.
🧰
Keep the provisioning tool current

Autologin failures during self-deploy are frequently traced back to an outdated provisioning tool. Use the latest version when you package it.

Docs: Deploy Teams Rooms app update tool

4 Create the Enrollment Status Page (ESP)

The ESP blocks device access until everything — policies, apps, certificates, the provisioning tool — has finished installing. I create an ESP profile and assign it to the MTR Dynamic device group.

Docs: Create a Windows Autopilot ESP profile

5 Create & assign the self-deploying Autopilot profile

This is the profile that makes the magic happen. I create a self-deploying mode Windows Autopilot profile — MTR_Windows_SelfDeploy — and assign it to the MTR Dynamic device group.

Self-deploying mode:

  • Joins the device to Microsoft Entra ID and enrolls it in Intune automatically.
  • Provisions all policies, apps, and profiles before anyone can touch the device.
  • Uses the device's TPM 2.0 for attestation — no user credentials entered during OOBE.
⚠️
Entra join only — not hybrid

Self-deploying mode is supported for Microsoft Entra join devices only. It is not supported for Entra hybrid join. Devices also need TPM 2.0 (a self-deploy attempt on a VM or non-TPM device can fail with the 0x800705B4 timeout).

Docs: Create & assign a self-deploying Autopilot profile

6 Create & assign a LAPS policy

The Local Administrator Password Solution (LAPS) manages and rotates the local admin password on the MTR device, and backs it up to Entra ID. An Entra/Intune admin can retrieve the current local-admin password from the portal — extremely handy for onsite troubleshooting (more on that in Section 12).

Password rotation settings

  • PasswordAgeDays controls the maximum local-admin password age. If you don't configure it, it defaults to 30 days.
  • For Microsoft Entra ID backup, the supported range is a minimum of 7 days and a maximum of 365 days.
☁️
Mark's recommendation

Set it and let it default to 30 daysLAPS auto-rotates the password on that cadence, so you always have a fresh, retrievable local-admin credential without managing it by hand.

Docs: LAPS management policy settings · LAPS for Teams Rooms

7 Create the resource account & assign Teams Rooms Pro

Every MTR device needs its own resource account — a room mailbox in Exchange Online that the device signs into and that users book in Outlook. For Windows-based MTR devices, the resource account UPN must start with the MTR- prefix. After you create the resource account, go into the M365 Admin Center and change the password.

Template resource account
MTR-Shure_Conference_Room@abc123.onmicrosoft.com
License
Microsoft Teams Rooms Pro
Naming convention
MTR- prefix (required for Windows MTR)

Create it cloud-only in Entra ID + Exchange Online, assign the Teams Rooms Pro license, and — importantly — do not require a password change at next sign-in (that breaks Teams Rooms sign-in).

⚠️
You never want to force a password change for a Teams Rooms resource account, this can cause issues

When you set the password, uncheck "Require this user to change their password when they first sign in." A forced change will cause the Teams Rooms device sign-in to fail.

Docs: Create resource account & assign the license

8 Configure the mailbox (PowerShell 7)

With the account created, I configure the mailbox for unattended room behavior using Exchange Online PowerShell, then set the time zone and disable password expiration via Microsoft Graph.

Connect to Exchange Online

Connect PowerShell 7
Import-Module ExchangeOnlineManagement
Connect-ExchangeOnline -UserPrincipalName Admin@abc123.onmicrosoft.com -ShowBanner:$false

Configure calendar processing (AutoAccept)

These are Microsoft's recommended Teams Rooms values: auto-accept bookings, keep the subject/body needed for join buttons, strip attachments the device can't open, and add a friendly response.

Set-CalendarProcessing Exchange Online
Set-CalendarProcessing -Identity "MTR-Shure_Conference_Room" `
  -AutomateProcessing AutoAccept `
  -AddOrganizerToSubject $false `
  -AllowRecurringMeetings $true `
  -DeleteAttachments $true `
  -DeleteComments $false `
  -DeleteSubject $false `
  -ProcessExternalMeetingMessages $true `
  -RemovePrivateProperty $false `
  -AddAdditionalResponse $true `
  -AdditionalResponse "Your meeting has been scheduled with the Shure, Microsoft Teams Room"

Set the time zone to Pacific Time for InfoComm 2026 in Las Vegas

For Las Vegas, use the Windows/Exchange time zone ID Pacific Standard Time. Even though the ID says “Standard,” it is the correct value for Pacific Time and automatically handles daylight saving time where applicable. Note: run the 2 PowerShell commands below individually.

Time zone Exchange Online
# Set
Set-MailboxRegionalConfiguration -Identity "MTR-Shure_Conference_Room@abc123.onmicrosoft.com" -TimeZone "Pacific Standard Time"

# Verify
Get-MailboxRegionalConfiguration -Identity "MTR-Shure_Conference_Room@abc123.onmicrosoft.com"

Turn off password expiration

Run elevated PowerShell 7 and connect to Microsoft Graph, then disable expiration so the password never expires, this makes sure no one gets locked out of the Teams Room due to a password reset.

Disable password expiration Microsoft Graph
Connect-MgGraph -Scopes "User.ReadWrite.All"

Update-MgUser -UserId MTR-Shure_Conference_Room@abc123.onmicrosoft.com `
  -PasswordPolicies "DisablePasswordExpiration"

Verify the password expiration action above worked - Expected -> PasswordPolicies:"DisablePasswordExpiration"

Run elevated PowerShell 7.

10 Exclude the room from MFA (CA bypass group)

Microsoft recommends that Teams Rooms resource accounts be excluded from MFA in Conditional Access — a shared room device can't satisfy an interactive MFA prompt. Rather than editing CA policies per account, I use a dynamic group.

  • Created an Entra security group: MTR-CA-Bypass, which is excluded from the MFA Conditional Access policies. (This group works for both Android and Windows MTR resource accounts.)
  • Membership is driven by the Department attribute on each resource account.
Dynamic membership rule, adds resource account to Entra Dynamic group, created above in step #2-> MTR-CA-BypassEntra ID
(user.department -eq "Resource Account for MTR")
☁️
Set the attribute, membership follows

As soon as a resource account is stamped with Department = "Resource Account for MTR", it auto-joins MTR-CA-Bypass and is excluded from MFA. The duplication script in Section 11 sets this attribute for you.

11 Duplicate the resource account with PowerShell

The first account, MTR-Shure_Conference_Room@abc123.onmicrosoft.com, was built by hand following the steps above — that's our template. For the second device — a Logitech Windows MTR (and every device after it), you can duplicate the template with a script so the configuration is identical every time.

Template (source)
MTR-Shure_Conference_Room@abc123.onmicrosoft.com
New account (target)
MTR-Microsoft_Express_Install_1@abc123.onmicrosoft.com
Belongs to (device 2)
Logitech Windows-based MTR

The script connects to Exchange Online + Microsoft Graph and reproduces the full template configuration on the new account:

  • Creates the Room mailbox (no forced password change).
  • Sets Usage Location and assigns the Teams Rooms Pro license.
  • Copies the template's calendar-processing (AutoAccept) settings.
  • Sets the mailbox time zone (Pacific Standard Time).
  • Disables password expiration.
  • Stamps Department = "Resource Account for MTR" so it auto-joins MTR-CA-Bypass.
📋
Prerequisites

PowerShell 7+ with the ExchangeOnlineManagement, Microsoft.Graph.Users, and Microsoft.Graph.Identity.DirectoryManagement modules. Run as Exchange Administrator + User Administrator (least privilege). Confirm -TeamsRoomsProSku matches the SkuPartNumber in your tenant.

Note, when you run the full script below, you will be asked to input the below 4 fields starting with "AdminUpn"

Duplicate one account PowerShell 7
.\Duplicate-MtrResourceAccount.ps1 `
    -AdminUpn       "Admin@abc123.onmicrosoft.com" `
    -TemplateUpn    "MTR-Shure_Conference_Room@abc123.onmicrosoft.com" `
    -NewUpn         "MTR-Microsoft_Express_Install_1@abc123.onmicrosoft.com" `
    -NewDisplayName "MTR-Microsoft_Express_Install_1"

The full script, you will be asked to input 4 fields(see above) and then enter a password for the resource account

Duplicate-MtrResourceAccount.ps1 PowerShell 7
<#
.SYNOPSIS
    Duplicates a Microsoft Teams Rooms (MTR) resource account from an existing
    "template" account so additional Windows-based MTR devices can be deployed
    quickly and consistently.

.DESCRIPTION
    Part 2 companion script for the blog series:
      "Deploying Microsoft Teams Rooms on Windows for InfoComm 2026"
    Author : Mark Layton  (Cloud Engineer)
    Field  : Jon          (Onsite Technician / handoff)

    Given a working template resource account (created by hand the first time,
    per the Part 2 walkthrough), this script stamps out an identical new
    resource account. It:

      1. Creates a new Room mailbox in Exchange Online.
      2. Sets a known password and disables "change password at next sign-in"
         (a forced change breaks Teams Rooms sign-in).
      3. Sets the Usage Location (required before a license can be assigned).
      4. Assigns the Microsoft Teams Rooms Pro license.
      5. Copies the template's calendar-processing (AutoAccept) settings.
      6. Sets the mailbox time zone.
      7. Disables password expiration (Update-MgUser).
      8. Stamps the Department attribute used by the "MTR-CA-Bypass" dynamic
         group so the account is auto-excluded from MFA Conditional Access.

    Re-running for the same account is safe: existing items are detected and
    skipped or updated rather than recreated.

    OFFICIAL DOCS CROSS-REFERENCED
      Create resource account ........ https://learn.microsoft.com/en-us/microsoftteams/rooms/create-resource-account
      Configure mailbox properties ... https://learn.microsoft.com/en-us/microsoftteams/hybrid-meetings-device-config-mailbox
      Conditional Access best practice  https://learn.microsoft.com/en-us/microsoftteams/rooms/conditional-access-and-compliance-for-devices
      Autopilot + Autologin .......... https://learn.microsoft.com/en-us/microsoftteams/rooms/autopilot-autologin

.NOTES
    Requires PowerShell 7+ and the modules:
        ExchangeOnlineManagement
        Microsoft.Graph.Users
        Microsoft.Graph.Identity.DirectoryManagement
    Run from an account with Exchange Administrator + User Administrator
    (or least-privilege equivalents). Avoid Global Admin unless required.

.EXAMPLE
    .\Duplicate-MtrResourceAccount.ps1 `
        -AdminUpn          "Admin@abc123.onmicrosoft.com" `
        -TemplateUpn       "MTR-Shure_Conference_Room@abc123.onmicrosoft.com" `
        -NewUpn            "MTR-Microsoft_Express_Install_1@abc123.onmicrosoft.com" `
        -NewDisplayName    "MTR-Microsoft_Express_Install_1" `
        -NewPassword       (Read-Host -AsSecureString "New room account password")

.EXAMPLE
    # Stamp out several rooms in one run (Install_1, Install_2, Install_3)
    1..3 | ForEach-Object {
        .\Duplicate-MtrResourceAccount.ps1 `
            -AdminUpn       "Admin@abc123.onmicrosoft.com" `
            -TemplateUpn    "MTR-Shure_Conference_Room@abc123.onmicrosoft.com" `
            -NewUpn         "MTR-Microsoft_Express_Install_$_@abc123.onmicrosoft.com" `
            -NewDisplayName "MTR-Microsoft_Express_Install $_"
    }
#>

[CmdletBinding(SupportsShouldProcess = $true)]
param(
    # The admin you connect to Exchange Online / Microsoft Graph as.
    [Parameter(Mandatory = $true)]
    [string] $AdminUpn,

    # The known-good account you already built by hand (the template).
    [Parameter(Mandatory = $true)]
    [string] $TemplateUpn = "MTR-Shure_Conference_Room@abc123.onmicrosoft.com",

    # The new resource account to create.
    [Parameter(Mandatory = $true)]
    [string] $NewUpn = "MTR-Microsoft_Express_Install_1@abc123.onmicrosoft.com",

    # Friendly display name shown in the GAL / Pro portal.
    [Parameter(Mandatory = $true)]
    [string] $NewDisplayName = "MTR-Microsoft_Express_Install_1",

    # Password for the new room account. Prompted securely if omitted.
    [Parameter(Mandatory = $false)]
    [System.Security.SecureString] $NewPassword,

    # Two-letter ISO usage location (required before licensing). Default: US.
    [Parameter(Mandatory = $false)]
    [string] $UsageLocation = "US",

    # Mailbox time zone (Windows time zone ID).
    # For Las Vegas / Pacific Time, use "Pacific Standard Time".
    [Parameter(Mandatory = $false)]
    [string] $TimeZone = "Pacific Standard Time",

    # SkuPartNumber for the Teams Rooms Pro license in your tenant.
    [Parameter(Mandatory = $false)]
    [string] $TeamsRoomsProSku = "Microsoft_Teams_Rooms_Pro",

    # Department value that the "MTR-CA-Bypass" dynamic group keys on.
    [Parameter(Mandatory = $false)]
    [string] $Department = "Resource Account for MTR"
)

$ErrorActionPreference = 'Stop'

function Write-Step  { param([string]$m) Write-Host "`n=== $m ===" -ForegroundColor Cyan }
function Write-Ok    { param([string]$m) Write-Host "  [OK]   $m" -ForegroundColor Green }
function Write-Skip  { param([string]$m) Write-Host "  [SKIP] $m" -ForegroundColor Yellow }

# ----------------------------------------------------------------------------
# 0. Prompt for the password if it was not supplied on the command line.
# ----------------------------------------------------------------------------
if (-not $NewPassword) {
    $NewPassword = Read-Host -AsSecureString "Enter password for $NewUpn"
}
# Plain-text copy used only for Exchange's New-Mailbox -Password / Graph reset.
$plainPwd = [System.Net.NetworkCredential]::new('', $NewPassword).Password
$alias    = ($NewUpn.Split('@')[0])   # local part, used as -Name / -Alias

# ----------------------------------------------------------------------------
# 1. Connect to Exchange Online and Microsoft Graph.
# ----------------------------------------------------------------------------
Write-Step "Connecting to Exchange Online and Microsoft Graph"

Import-Module ExchangeOnlineManagement -ErrorAction Stop
Connect-ExchangeOnline -UserPrincipalName $AdminUpn -ShowBanner:$false
Write-Ok "Connected to Exchange Online as $AdminUpn"

Import-Module Microsoft.Graph.Users -ErrorAction Stop
Import-Module Microsoft.Graph.Identity.DirectoryManagement -ErrorAction Stop
Connect-MgGraph -Scopes "User.ReadWrite.All","Organization.Read.All" -NoWelcome
Write-Ok "Connected to Microsoft Graph"

# ----------------------------------------------------------------------------
# 2. Create the new Room mailbox/resource account (skip if it already exists).
# ----------------------------------------------------------------------------
Write-Step "Creating room mailbox: $NewUpn"

$existing = Get-Mailbox -Identity $NewUpn -ErrorAction SilentlyContinue

if ($existing) {
    Write-Skip "Mailbox already exists - will update settings only."
}
else {
    if ($PSCmdlet.ShouldProcess($NewUpn, "New-Mailbox -Room")) {

        New-Mailbox -Room `
            -Name $alias `
            -Alias $alias `
            -DisplayName $NewDisplayName `
            -MicrosoftOnlineServicesID $NewUpn `
            -PrimarySmtpAddress $NewUpn `
            -EnableRoomMailboxAccount $true `
            -RoomMailboxPassword $NewPassword | Out-Null

        Write-Ok "Room mailbox created."
    }
}

# ----------------------------------------------------------------------------
# 3. Set Usage Location (required before a license can be assigned).
# ----------------------------------------------------------------------------
Write-Step "Setting Usage Location ($UsageLocation) and Department"

# Graph user objects can lag mailbox creation by a few seconds.
$graphUser = $null
for ($i = 0; $i -lt 12 -and -not $graphUser; $i++) {
    $graphUser = Get-MgUser -Filter "userPrincipalName eq '$NewUpn'" -ErrorAction SilentlyContinue
    if (-not $graphUser) { Start-Sleep -Seconds 5 }
}
if (-not $graphUser) { throw "User $NewUpn not yet visible in Microsoft Graph. Re-run in a minute." }

Update-MgUser -UserId $graphUser.Id -UsageLocation $UsageLocation -Department $Department
Write-Ok "Usage Location = $UsageLocation; Department = '$Department' (drives MTR-CA-Bypass group)."

# ----------------------------------------------------------------------------
# 4. Assign the Microsoft Teams Rooms Pro license.
# ----------------------------------------------------------------------------
Write-Step "Assigning Teams Rooms Pro license"

$sku = Get-MgSubscribedSku -All | Where-Object SkuPartNumber -eq $TeamsRoomsProSku
if (-not $sku) {
    Write-Skip "SKU '$TeamsRoomsProSku' not found. Available SKUs:"
    Get-MgSubscribedSku -All |
        Select-Object SkuPartNumber, SkuId,
            @{n='Avail';e={ $_.PrepaidUnits.Enabled - $_.ConsumedUnits }} |
        Format-Table -AutoSize
    throw "Adjust -TeamsRoomsProSku to match a SkuPartNumber listed above, then re-run."
}

$hasLicense = (Get-MgUserLicenseDetail -UserId $graphUser.Id -ErrorAction SilentlyContinue).SkuId -contains $sku.SkuId
if ($hasLicense) {
    Write-Skip "Teams Rooms Pro already assigned."
}
else {
    Set-MgUserLicense -UserId $graphUser.Id `
        -AddLicenses @(@{ SkuId = $sku.SkuId }) `
        -RemoveLicenses @() | Out-Null
    Write-Ok "Teams Rooms Pro ($($sku.SkuPartNumber)) assigned."
}

# ----------------------------------------------------------------------------
# 5. Copy calendar-processing (AutoAccept) settings from the template.
#    Falls back to Microsoft's recommended MTR values if the template can't
#    be read, so the new room always lands in a known-good state.
# ----------------------------------------------------------------------------
Write-Step "Configuring calendar processing (copying from template)"

$tpl = Get-CalendarProcessing -Identity $TemplateUpn -ErrorAction SilentlyContinue

$calParams = @{
    Identity                      = $NewUpn
    AutomateProcessing            = if ($tpl) { $tpl.AutomateProcessing }            else { 'AutoAccept' }
    AddOrganizerToSubject         = if ($tpl) { $tpl.AddOrganizerToSubject }         else { $false }
    AllowRecurringMeetings        = if ($tpl) { $tpl.AllowRecurringMeetings }        else { $true }
    DeleteAttachments             = if ($tpl) { $tpl.DeleteAttachments }             else { $true }
    DeleteComments                = if ($tpl) { $tpl.DeleteComments }                else { $false }
    DeleteSubject                 = if ($tpl) { $tpl.DeleteSubject }                 else { $false }
    ProcessExternalMeetingMessages= if ($tpl) { $tpl.ProcessExternalMeetingMessages} else { $true }
    RemovePrivateProperty         = if ($tpl) { $tpl.RemovePrivateProperty }         else { $false }
    AddAdditionalResponse         = if ($tpl) { $tpl.AddAdditionalResponse }         else { $true }
    AdditionalResponse            = if ($tpl -and $tpl.AdditionalResponse) { $tpl.AdditionalResponse }
                                    else { "This is a Microsoft Teams Meeting room" }
}
Set-CalendarProcessing @calParams
if ($tpl) { Write-Ok "Calendar processing copied from $TemplateUpn." }
else      { Write-Ok "Template not readable - applied Microsoft recommended MTR defaults." }

# ----------------------------------------------------------------------------
# 6. Set the mailbox time zone (and language).
# ----------------------------------------------------------------------------
Write-Step "Setting mailbox time zone: $TimeZone"
Set-MailboxRegionalConfiguration -Identity $NewUpn -Language 1033 -TimeZone $TimeZone
$rc = Get-MailboxRegionalConfiguration -Identity $NewUpn
Write-Ok "Time zone = $($rc.TimeZone)"

# ----------------------------------------------------------------------------
# 7. Disable password expiration so the room never gets locked out.
# ----------------------------------------------------------------------------
Write-Step "Disabling password expiration"
Update-MgUser -UserId $graphUser.Id -PasswordPolicies "DisablePasswordExpiration"
Write-Ok "PasswordPolicies = DisablePasswordExpiration"

# ----------------------------------------------------------------------------
# 8. Summary / handoff notes for the onsite technician.
# ----------------------------------------------------------------------------
Write-Step "DONE - Resource account ready for Autologin"
Write-Host @"
  Account ......... $NewUpn
  Display name .... $NewDisplayName
  License ......... Teams Rooms Pro
  Time zone ....... $TimeZone
  CA bypass ....... Department = '$Department' (auto-joins MTR-CA-Bypass)
  Password ........ (the value you supplied at the prompt)

  NEXT STEPS
   1. Verify sign-in with the Teams Rooms sign-in test:
      https://testconnectivity.microsoft.com/tests/TeamsMTRDeviceSignIn/input
   2. In the Teams Rooms Pro Management portal, assign this resource account
      to the Autopilot-registered Windows MTR device (for Autologin).
   3. Hand the UPN + password to Jon (onsite tech) only if a manual sign-in
      is ever needed - Autologin should make that unnecessary.
"@ -ForegroundColor Green

# Disconnect-ExchangeOnline -Confirm:$false
# Disconnect-MgGraph

12 Remote access & the power of Autologin

Once the Shure Windows OS MTR device is registered, Entra ID grouped, and tied to its resource account, deployment onsite is ready to go: Jon the onsite tech unboxes it, plugs in power and Ethernet, powers it on, enters the resource account user and password along with the MFA bypass("Skip this step"). The device provisions Windows, Entra-joins, enrolls in Intune, updates the Teams Rooms app, and Autologs in to its assigned resource account — touchless, in minutes.

A real-world save (the Lenovo story)

On a prior Windows MTR deployment, the onsite tech (no global admin, no local admin rights) couldn't get into the device's local admin. We even pulled in Lenovo support and ended up doing a factory reset — and the local-admin problem still persisted onsite.

What ultimately solved it: I logged into the MTR device remotely from the US to Canada — over 3,200 km away — using my global admin login, reached the settings through the Teams Rooms Pro portal, changed the local admin password, and handed that off to the onsite tech. Being able to complete (and recover) the configuration remotely, after the Teams Room completed Autopilot/Autologin, was genuinely powerful — the remote login and admin experience in the Teams Rooms Pro portal is excellent.

☁️
LAPS makes this even cleaner

With the LAPS policy from Section 6, the current local-admin password is retrievable from the Entra/Intune portal — so the next time an onsite tech needs local admin, I can pull it without a factory reset or a vendor support ticket.
